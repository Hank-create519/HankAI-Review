# 瀚海贸易 Agent 系统 · 全新架构设计

> Hanhai Trade Agent System — Ground-Up Architecture
> 从零设计，不复用审查系统的三层辩论框架

---

## 一、为什么不用旧架构

审查系统的本质是「多个AI从不同角度审视同一份材料→交叉辩论→出报告」。它的基因是**同质化角色的对抗性协作**。

贸易系统的本质是「多个专业化Agent分工完成一条业务流水线」。它的基因是**异质化角色的接力式协作**。

强行改造只会绑手绑脚。重做。

## 三、8个Agent角色详细定义

### 3.1 选品工（Scout）
**一句话**：在全球市场里找到「国内便宜、国外贵」的东西。

| 项目 | 内容 |
|---|---|
| 触发 | 用户指定品类 / 定时自动扫描 / 新机会发现 |
| 输入 | 品类关键词、预算上限、目标平台 |
| 核心逻辑 | 爬1688/拼多多/义乌购价格 → 比对eBay/Shopee/Amazon售价 → 计算价差率 |
| 输出 | 候选商品列表写入黑板 `商品池`，每项含：品名、1688链接、海外参考价、预估利润、风险标记 |
| 工具 | web_search、web_fetch |
| 审核 | 候选列表需用户勾选确认后才进入定价环节 |

### 3.2 定价工（Pricer）
**一句话**：算清一笔生意到底能赚多少钱。

| 项目 | 内容 |
|---|---|
| 触发 | 黑板 `商品池` 中出现新候选商品 |
| 输入 | 商品信息（成本价、重量、尺寸）+ 目标平台 |
| 核心逻辑 | 逐项累加：货价+国内运费+国际运费(按重量查价)+平台佣金+支付手续费+关税估算+退货预留5% |
| 输出 | 写入黑板：建议售价区间、各项成本拆解、预期利润率 |
| 审核 | 用户可调整利润率目标 |

**成本公式（内置）**：
```
总成本 = 商品进价
       + 国内运费（默认8元/kg，可配置）
       + 国际运费（查物流商报价表）
       + 平台佣金（eBay 13.25% / Shopee 6% / Amazon 15%）
       + PayPal手续费（4.4% + $0.30）
       + 关税预估（目的国税率 × 申报价值，默认按10%估算）
       + 退货损耗预留（售价 × 5%）
建议售价下限 = 总成本
建议售价 = 总成本 × (1 + 目标利润率)
```

### 3.3 上架工（Lister）
**一句话**：把商品挂到海外平台，让外国人能看到、想买。

| 项目 | 内容 |
|---|---|
| 触发 | 定价完成，用户确认上架 |
| 输入 | 商品详情 + 定价方案 + 目标平台 |
| 输出 | 结构化listing草稿（标题+描述+关键词+图片建议），写入黑板 |
| 阶段1（当前） | 生成草稿文本，用户手动复制粘贴上架 |
| 阶段2（远期） | Playwright浏览器自动化，模拟人工上架操作 |

**Listing生成规则**：
- 标题：`[核心词] + [属性词] + [场景词] + [卖点词]`，60-80字符
- 描述：Markdown格式，含规格表+使用场景+售后政策
- 关键词：提取高搜索量长尾词，填入item specifics
- 图片建议：从商品页提取或提示用户准备

### 3.4 采购工（Buyer）
**一句话**：海外有人下单，立刻帮你在国内下单发货。

| 项目 | 内容 |
|---|---|
| 触发 | 黑板 `订单池` 出现新订单（目前为手动录入，远期对接平台API） |
| 输入 | 订单商品SKU + 客户地址 + 供应商链接 |
| 输出 | 国内采购订单信息写入黑板 |
| 阶段1 | 生成下单清单（供应商、商品、数量、收货地址），用户手动下单 |
| 阶段2 | 模拟1688/拼多多下单流程（浏览器自动化） |
| 风险控制 | 下单前自动检查：价格是否变化（>10%报警）、库存状态、供应商评分 |

### 3.5 物流工（Tracker）
**一句话**：盯着包裹从中国工厂到外国客户手里，中间不出岔子。

| 项目 | 内容 |
|---|---|
| 触发 | 采购工完成下单，拿到物流单号 |
| 输入 | 物流单号 + 承运商 |
| 核心逻辑 | 定时查询物流状态（国内段+国际段），识别异常 |
| 输出 | 写入黑板：物流时间线、预计到达日、异常标记 |
| 告警 | 超48h无更新→通知 / 延迟>3天→生成客户安抚模板 / 签收→触发售后跟进 |

### 3.6 客服工（Support）
**一句话**：客户找上门来，先挡第一波，搞不定的再找你。

| 项目 | 内容 |
|---|---|
| 触发 | 手动输入客户消息 / 远期对接平台消息 |
| 输入 | 客户消息 + 订单上下文（从黑板读取） |
| 分级处理 | L1物流查询→自动回复 / L2质量问题→生成方案模板，用户审核 / L3纠纷退款→立即升级，附完整历史 |
| 输出 | 回复草稿 → 用户确认 → 发送（远期自动） |

### 3.7 财务工（Finance）
**一句话**：帮你记账，告诉你这个月到底赚了多少。

| 项目 | 内容 |
|---|---|
| 触发 | 每笔订单完成后 / 用户要求查看报表 |
| 输入 | 订单全链路数据（从黑板读取） |
| 输出 | 利润表、现金流表、按平台/品类/时间维度的分析 |
| 能力 | 自动生成月报 / 税务提醒（月销接近起征点时预警） |

### 3.8 风控工（Risk）
**一句话**：在所有环节里找可能让你亏钱或被封号的风险。

| 项目 | 内容 |
|---|---|
| 触发 | 每个Agent执行后自动运行 / 用户主动请求 |
| 检查项 | 品牌侵权检查 / 禁运品类检查 / 价格异常波动 / 供应商信用评分 / 平台规则红线 |
| 输出 | 风险报告写入黑板，高风险项自动阻止后续流程并通知用户 |

## 四、编排引擎设计

### 4.1 执行模型：事件驱动的DAG

不再用固定的三层流水线，而是由编排引擎根据用户意图动态生成执行DAG。

**示例：用户说「帮我在1688找5个利润>30%的手机壳，上架到eBay」**

编排引擎生成的DAG：
```
           ┌──────────┐
           │  意图理解  │
           └─────┬────┘
                 ▼
           ┌──────────┐
           │  选品工    │ → 搜索1688 + 对比eBay价格
           └─────┬────┘
                 ▼
        ┌────────────────┐
        │  人工审核网关    │ ← 确认候选商品
        └───────┬────────┘
                ▼
           ┌──────────┐
           │  定价工    │ → 逐项计算成本+利润
           └─────┬────┘
                ▼
           ┌──────────┐
           │  风控工    │ → 品牌侵权/禁运检查
           └─────┬────┘
                ▼
        ┌────────────────┐
        │  人工审核网关    │ ← 确认定价方案
        └───────┬────────┘
                ▼
           ┌──────────┐
           │  上架工    │ → 生成eBay listing草稿
           └─────┬────┘
                ▼
        ┌────────────────┐
        │  人工审核网关    │ ← 确认listing内容
        └───────┬────────┘
                ▼
           ┌──────────┐
           │  任务完成  │
           └──────────┘
```

### 4.2 DAG节点的四种类型

| 节点类型 | 说明 | 示例 |
|---|---|---|
| Agent节点 | 调用一个Agent执行任务 | 选品工、定价工 |
| 审核节点 | 暂停等待用户确认 | 确认候选商品列表 |
| 条件节点 | 根据条件分支 | 利润率>30%继续，否则跳过 |
| 并行节点 | 多个Agent同时执行 | 同时上架eBay+Shopee |

### 4.3 编排引擎核心能力

- **动态DAG生成**：根据用户指令自动规划执行路径
- **节点状态管理**：pending → running → completed / failed / paused
- **审核网关**：在关键决策点暂停，弹出选项让用户确认
- **失败重试**：Agent执行失败自动重试（最多2次），失败后升级给用户
- **超时处理**：Agent执行超过阈值自动终止并通知
- **并行调度**：无依赖关系的节点同时执行

## 五、黑板模式

### 5.1 设计原则

Agent之间不直接通信。所有信息通过黑板共享。这带来三个好处：

1. **解耦**：增加/删除Agent不影响其他Agent
2. **可追溯**：任何时刻的黑板状态都是完整快照，方便调试
3. **可中断恢复**：系统崩溃后从黑板恢复状态继续执行

### 5.2 黑板数据结构

```typescript
interface Blackboard {
  // 任务元信息
  taskId: string;
  status: 'pending' | 'running' | 'paused' | 'completed' | 'failed';
  currentNode: string;

  // 商品池
  productPool: Product[];

  // 订单池
  orderPool: Order[];

  // 用户配置
  config: {
    targetPlatforms: string[];     // ['ebay', 'shopee']
    targetProfitRate: number;      // 0.30
    maxBudgetPerItem: number;      // 单件预算上限
    riskTolerance: 'low' | 'medium' | 'high';
    shippingFrom: string;          // 发货地
    shippingTo: string[];          // 目标国家
  };

  // 执行日志
  log: LogEntry[];

  // 当前任务的中间结果
  currentDAG: DAGNode[];
  humanDecisions: HumanDecision[];
}

interface Product {
  id: string;
  name: string;
  sourceUrl: string;           // 1688链接
  sourcePrice: number;         // 国内进价
  weight: number;               // 重量(g)
  dimensions: { l: number; w: number; h: number };
  overseasRefPrice: number;    // 海外参考售价
  profitMargin: number;         // 预估利润率
  riskFlags: string[];          // 风险标记
  status: 'candidate' | 'approved' | 'priced' | 'listed' | 'rejected';
  listingUrl?: string;          // 上架后的链接
}

interface Order {
  id: string;
  product: Product;
  customerName: string;
  customerAddress: string;
  salePrice: number;
  platform: string;
  supplierOrderId?: string;     // 国内采购单号
  trackingNumber?: string;
  trackingHistory?: TrackingEvent[];
  status: 'pending' | 'purchased' | 'shipped' | 'delivered' | 'dispute';
  profit?: number;              // 实际利润（完成后计算）
}

interface LogEntry {
  timestamp: string;
  agent: string;
  action: string;
  result: string;
  level: 'info' | 'warn' | 'error';
}
```
