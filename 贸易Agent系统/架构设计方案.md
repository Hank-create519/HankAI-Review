# 瀚海贸易 Agent 团队 · 架构设计方案

> Hanhai Trade Agent Team — Architecture Design
> 基于 HankAI 审查系统 V1.0 三层辩论框架改造

---

## 一、项目定位

将 HankAI 审查系统的「多层 AI 辩论引擎」改造为「自动化跨境贸易 Agent 团队」。

**核心业务流**：
```
国内低价平台选品 → 定价分析（含运费/关税/平台费） → 海外平台上架 → 
接单 → 自动向国内供应商下单 → 物流追踪 → 售后处理 → 利润核算
```

**与审查系统的本质差异**：

| 维度 | V1.0 审查系统 | 贸易 Agent 系统 |
|------|-------------|----------------|
| 任务性质 | 纯分析（只读） | 分析+执行（读写） |
| 交互对象 | 文本输入→报告输出 | 多平台API/网页操作 |
| 实时性要求 | 低（几分钟出报告） | 中（订单需及时处理） |
| 外部依赖 | LLM API | LLM + 电商平台 + 支付 + 物流 |
| Agent 角色 | 3个同质辩论AI | 多个异构职能Agent |

### 2.2 各角色详细定义

#### 协调者 Agent（Orchestrator）
- **职责**：接收用户指令 → 拆解为子任务 → 分发给职能Agent → 汇总结果 → 异常时升级给用户
- **输入**：用户自然语言指令（如「帮我找5个利润>30%的数码配件，上架到eBay」）
- **输出**：任务执行报告 + 待用户确认的决策点
- **关键能力**：任务DAG编排、进度追踪、超时处理、人类中断接入

#### 选品 Agent（Product Scout）
- **职责**：扫描1688/拼多多/义乌购等平台，结合海外eBay/Shopee/Amazon价格，发现有利可图的商品
- **输入**：品类范围、预算、目标利润率
- **输出**：候选商品清单（含国内价、海外参考价、预估利润、竞争度）
- **工具**：web_search / web_fetch / 平台API（远期）
- **决策逻辑**：
  - 价差率 = (海外售价 - 国内成本 - 预估运费) / 国内成本
  - 竞争度 = 海外平台同类商品数量 / 评分分布
  - 合规检查 = 是否涉及品牌侵权 / 禁运品类

#### 定价 Agent（Pricing Analyst）
- **职责**：综合计算商品总成本，给出海外平台建议售价和预期利润
- **输入**：商品信息（成本价、重量、尺寸）+ 目标平台
- **输出**：定价方案（含各项成本拆解）
- **成本模型**：
  ```
  总成本 = 商品成本 + 国内运费 + 国际运费 + 平台佣金 + 支付手续费 + 关税预估 + 退货预留
  建议售价 = 总成本 × (1 + 目标利润率)
  实际利润 = 售价 - 总成本
  ```
- **继承V1.0辩论机制**：此处可启用多方校验模式（见第四节）

#### 上架 Agent（Listing Manager）
- **职责**：在海外平台（eBay/Shopee等）创建商品listing，包括标题优化、描述撰写、图片处理、价格设置
- **输入**：商品详情 + 定价方案 + 目标平台
- **输出**：已发布的listing链接
- **实现方式**：
  - 短期：输出结构化listing草稿（标题/描述/关键词/图片），用户手动粘贴上架
  - 中期：通过平台API自动上架
  - 长期：浏览器自动化（Playwright/Puppeteer）模拟人工操作
- **多平台适配**：每个平台独立配置模板（eBay item specifics / Shopee variation / Amazon bullet points）

#### 采购 Agent（Procurement Agent）
- **职责**：收到海外订单后，自动向国内供应商下单（1688/拼多多等），填写海外客户地址为收货地址
- **输入**：订单详情 + 供应商链接
- **输出**：国内订单号 + 物流单号
- **风险控制**：
  - 下单前核对商品SKU、价格、库存
  - 异常价格波动（涨价>10%）自动暂停并通知用户
  - 供应商评分过低自动预警

#### 物流追踪 Agent（Logistics Tracker）
- **职责**：追踪国内→国际全链路物流状态，异常自动预警
- **输入**：物流单号 + 承运商
- **输出**：物流状态时间线 + 预计到达时间
- **告警规则**：
  - 超过48小时无物流更新 → 通知用户
  - 预计到达延迟超过3天 → 自动生成给客户的解释模板
  - 包裹签收 → 触发售后Agent启动评价跟进

#### 客服 Agent（Customer Service）
- **职责**：处理客户咨询、纠纷、退换货请求
- **输入**：客户消息 + 订单上下文
- **输出**：回复草稿（用户审核后发送）或自动回复（低风险场景）
- **分级处理**：
  - L1（物流查询）：自动回复，无需审核
  - L2（产品质量问题）：生成回复模板 + 可能的解决方案，用户审核
  - L3（纠纷/退款）：立即升级给用户，附带完整订单历史和建议

#### 财务 Agent（Finance Tracker）
- **职责**：记录每笔交易的收支，定期生成利润报表
- **输入**：订单数据 + 各项费用
- **输出**：利润表 / 现金流表 / 税务预估
- **数据维度**：按商品 / 平台 / 时间段 / 品类 分类统计
- **税务提示**：月度销售额接近起征点时自动提醒

### 3.2 核心创新：双轨架构

V1.0的问题是「所有任务都走辩论」，导致简单任务浪费token。

本架构采用**双轨制**：

| | 执行轨道 | 辩论轨道 |
|---|---|---|
| **触发条件** | 所有任务默认走 | 仅关键决策节点触发 |
| **参与角色** | 单一职能Agent | 3个辩论AI（继承V1.0） |
| **触发节点举例** | 搜索商品、生成文案 | 定价合理性、纠纷处理方案、选品风险评估 |
| **Token 消耗** | 低（单次调用） | 中（3×N轮调用） |
| **输出** | 直接结果 | 共识+分歧+建议 |

| 层 | 技术 | 说明 |
|---|---|---|
| Agent 运行时 | TypeScript | 继承V1.0的ReviewEngine改造为AgentRuntime |
| 前端 | React 19 + Tailwind CSS 4 | 继承V1.0的UI框架 |

## 四、辩论机制的贸易场景适配

### 4.1 辩论触发场景

不所有任务都辩论，仅在以下三类场景启用：

#### 场景A：定价合理性辩论
```
触发条件：定价Agent输出方案后
辩论角色：
  - 激进定价AI：主张高价策略（利润最大化）
  - 保守定价AI：主张低价策略（竞争力优先）  
  - 平衡定价AI：综合两方，给出最优区间
输出：最终建议售价区间 + 双方论据摘要
```

#### 场景B：选品风险辩论
```
触发条件：选品Agent发现潜在风险商品（如涉及品牌擦边、价格波动大）
辩论角色：
  - 乐观评估AI：论证风险可控
  - 审慎评估AI：论证风险不可忽视
  - 裁决AI：给出最终风险评估和建议
输出：风险等级 + 是否推荐 + 替代方案
```

#### 场景C：纠纷处理辩论
```
触发条件：客服Agent遇到退货/退款纠纷
辩论角色：
  - 客户立场AI：从客户角度论证诉求合理性
  - 商家立场AI：从商家角度论证拒绝理由
  - 调解AI：给出双赢方案
输出：建议处理方案 + 客户回复模板 + 最大让步底线
```

### 4.2 辩论层与执行层的衔接

```
执行轨道输出 → 触发条件判断 → 如需辩论：
  1. 构建辩论上下文（争议点+背景数据）
  2. 启动3方辩论（复用V1.0引擎，1-3轮）
  3. 阶段汇总AI压缩输出
  4. 将辩论结果写回黑板
  5. 协调者Agent读取黑板，继续执行或暂停等用户决策
```

## 六、关键设计决策

### 6.1 为什么保留三层架构
V1.0的三层（理解→执行→报告）本质是合理的任务处理范式。贸易场景只是把「辩论AI审查文档」改为「职能Agent执行任务+关键节点辩论」，架构思想不变。

### 6.2 为什么加入双轨制
全辩论太慢太贵，零辩论太蠢太险。双轨制在效率和安全之间取得平衡——日常操作走快速执行轨道，关键决策走辩论轨道。

### 6.3 为什么先从「生成草稿」开始而非「自动操作」
- 自动上架/下单涉及资金安全，需要用户审核环节
- 平台反爬虫机制可能导致账号风险
- 渐进式自动化降低初期开发复杂度
- 用户可以在使用中逐步建立信任

### 6.4 与V2.0的关系
本系统是V2.0规划中「通用Multi-Agent编排引擎」的**首个垂直领域实例**。贸易Agent的职能角色和双轨架构可以视为DAG编排引擎的具体应用。两者可以并行开发，贸易系统复用V2.0的基建能力（electron-vite、DB迁移、多Provider等）。
